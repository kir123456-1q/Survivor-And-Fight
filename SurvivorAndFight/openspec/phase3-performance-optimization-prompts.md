## 第 3 期：性能与大规模实体优化 – 提示词文档

### 总体规划提示词

```text
请作为熟悉 Web 性能与 ECS 优化的工程师，帮我为“ECS 性能与大规模实体优化”设计详细方案（不写实现代码）。

当前状态：
- 已有 ECS 核心、类吸血鬼幸存者核心玩法（玩家、敌人、战斗、升级）。
- 目前实体规模中等，尚未针对“大量实体 + 60 FPS”做专门优化。

目标：
- 在 Web 浏览器中，支撑“同屏数千个敌人/子弹”的场景，并尽量保持 60 FPS（至少提供合理的优化路径和测量方法）。  

请输出内容包括：
1. **性能瓶颈分析假设**  
   - 从 ECS 视角推演可能的性能瓶颈（例如：组件遍历、缓存未命中、频繁 GC、渲染过多 Draw Call 等）。  
   - 按优先级列出“最可能痛点 → 次要痛点”。

2. **优化策略总览**  
   - 组件存储与访问模式优化（例如 SoA、稠密数组）。  
   - 空间划分 + 视锥体裁剪策略（如 2D 网格分区，对超出视野或距离较远实体降级更新）。  
   - LOD 策略（减少更新频率或仅逻辑/不渲染）。  
   - 对象池与内存管理（控制 GC 峰值）。  

3. **验证与监控方案**  
   - 如何构造“极端压力场景”来测性能（怪物/子弹数量、行为简化）。  
   - 需要收集哪些指标（FPS、总实体数、系统耗时分布等），准备在第 4 期接入工具链。  

4. **分阶段实施顺序建议**  
   - 推荐的实施顺序（例如：先测量 → 调整组件存储 → 加入空间划分 → 引入 LOD → 优化对象池）。  

请结构化输出，方便后续拆成一个个实现子任务。
```

### 子任务提示词：组件存储与迭代优化设计

```text
只关注“组件存储结构与遍历方式优化”，请在不写代码实现的前提下，详细设计：

1. **现状假设与问题分析**  
   - 假设当前组件存储为“对象数组 + Map 查询”等较直观写法，说明这种方式在上千实体时的典型问题（缓存局部性差、GC 压力等）。  

2. **目标存储模型设计（建议）**  
   - 设计一种更 Cache 友好的方案（例如：  
     - 每种组件类型维护一个稠密数组 `components[]`，配合 `entityId -> index` 映射；  
     - 或更接近 archetype 的简化布局），  
     并描述该方案的核心思想和时间/空间复杂度特点。  

3. **迁移策略**  
   - 如何从现有实现逐步迁移到新存储：  
     - 哪些公共接口尽量保持不变；  
     - 哪些地方需要重构调用方式。  
   - 迁移过程中如何验证正确性与性能改进（对比测试场景设计）。

请给出接口层面的 TypeScript 伪类型/方法签名，但不要写实现细节。
```

### 子任务提示词：空间划分、视锥体裁剪与 LOD 设计

```text
专注于“空间划分 + 视锥体裁剪 + LOD”，请设计一个适用于 2D/俯视视角 H5 游戏的方案：

1. **空间划分结构设计**  
   - 推荐使用何种结构（例如：均匀网格 vs 四叉树），并说明选择理由（实现复杂度 vs 性能收益）。  
   - 数据结构草图：网格单元存什么（实体 ID 列表？），如何更新实体所在格子。

2. **视锥体/范围裁剪策略**  
   - 如何根据摄像机/玩家位置和视野尺寸，快速判断哪些格子/实体“需要完整更新与渲染”。  
   - 对不在视野内的实体，采用什么策略（完全跳过更新、降低更新频率、仅执行关键逻辑等）。

3. **LOD 策略设计**  
   - 设计 2–3 个 LOD 等级（例如：近/中/远），说明每一档的更新与渲染策略。  
   - LOD 与空间划分如何协同（例如，根据距离/网格层级决定 LOD）。

请使用伪代码或流程图式文字说明整体流程，重点在“如何在 ECS 系统层面插入这些逻辑”，不写实际代码。
```

## 第 4 期：工具链与高级玩法扩展 – 提示词文档

### 总体规划提示词

```text
请作为工程化导向的游戏技术负责人，帮我为“性能分析工具链 + 高级玩法扩展”设计详细方案（不写代码）。

当前状态：
- ECS 核心 + 类吸血鬼幸存者玩法已基本可玩。  
- 第 3 期已做了部分性能优化，存在一定规模的实体压力场景。  

本期目标：
- 建立轻量级的“运行时性能与状态可视化工具链”。  
- 增强玩法层面的进度系统（关卡/波次/Boss）和演示性。  

请输出内容包括：
1. **工具链需求设计**  
   - 运行时需要展示的关键指标（FPS、系统耗时、实体/组件数量、对象池使用情况等）。  
   - Debug UI 大致形态（简单面板/开关），如何在不影响正常玩家体验的前提下使用。  
   - 收集这些指标所需的“打点 API 设计思想”。

2. **高级玩法扩展设计**  
   - 关卡与波次系统：  
     - 关卡/波次配置结构（字段、示例）。  
     - 驱动系统如何根据时间/玩家状态推进波次和 Boss 出场。  
   - 演示/教程场景：  
     - 为教学 ECS 使用设计一个简单场景（例如：新增一个技能或敌人类型的完整流程）。

3. **面向团队复用的工程化建议**  
   - 如何组织文档/示例，使这套 ECS 框架+玩法原型适合复用到其他项目。  
   - 哪些地方需要保持接口稳定，哪些可以仅在示例项目中存在。

请结构化输出，便于直接拆分成若干实现子任务。
```

### 子任务提示词：运行时性能监控与 Debug UI 设计

```text
只聚焦“运行时性能监控 + Debug UI 面板”，请设计：

1. **指标与打点 API 设计**  
   - 列出需要监控的指标（FPS、每个 System 的平均/最大耗时、实体总数、各类实体数量、对象池使用率等）。  
   - 为这些指标设计一套简单的打点 API 思路（例如 `Profiler.beginSystem(name)` / `Profiler.endSystem(name)`），接口用 TypeScript 伪代码说明即可。

2. **数据采样与开销控制**  
   - 说明如何控制打点本身的性能开销（例如采样频率、滑动平均窗口）。  
   - 在 Release/Debug 模式下如何切换（例如通过编译宏/配置开关）。

3. **Debug UI 面板结构**  
   - 面板上展示哪些区块（FPS 概览、系统耗时表、实体统计、对象池统计）。  
   - 在 Laya 场景中如何叠加一个简单 UI（不需要具体实现，只描述层级和交互方式）。

请用条目和简单伪接口表示，不写完整实现。
```

### 子任务提示词：关卡、波次与 Boss 系统设计

```text
专注设计“关卡/波次/Boss 系统”，在 ECS 框架下回答：

1. **数据驱动配置结构**  
   - 设计一个关卡配置和波次配置的 TypeScript 接口（伪代码即可），包含：  
     - 关卡持续时间或胜利条件。  
     - 每一波的开始时间/持续时间、敌人种类与数量、生成区域等。  
     - Boss 出场条件和参数。

2. **驱动系统流程**  
   - 描述一个 WaveSystem/StageSystem 的更新流程：如何根据时间或玩家状态推进波次。  
   - Boss 来临时如何切换音乐、表现或难度（可以只描述事件/回调机制，不做具体表现）。

3. **与现有玩法/系统的对接**  
   - 如何复用现有的敌人生成与 AI 系统，只通过配置驱动出不同的玩法节奏。  
   - 如何向 Debug 工具链输出关键状态（当前关卡、当前波次、剩余敌人数量等）。

请用流程和接口伪代码说明，方便后续实现。
```
